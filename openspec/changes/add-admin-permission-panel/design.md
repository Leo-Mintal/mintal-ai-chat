## Context

当前应用已经具备登录、用户资料与模型列表等能力，但没有单独的后台权限分发入口。管理员在给账号授予调用额度时缺少可视化操作面板，且现有交互风格偏“卡片+玻璃拟态”，新增后台页面必须保持同一视觉体系，避免出现风格断层。

本次改动会横跨页面路由/状态管理、侧边栏入口、表单交互、前端本地鉴权闸口与配额提交服务调用。后台页面需要通过管理员密码闸口（默认 `liao123...`）后才能进入，并提供“用户名 + 调用次数（默认 10）”授权能力。

## Goals / Non-Goals

**Goals:**
- 新增后台管理页面并复用现有 UI 组件与样式系统，保持一致的视觉语言。
- 新增管理员密码闸口，阻止未授权用户直接操作配额授权。
- 提供用户配额授权表单：用户名、调用次数（默认 10），并具备输入校验、提交反馈。
- 预留与后端接口对接的服务层调用点，便于后续替换为真实权限分发 API。

**Non-Goals:**
- 不在本次改动中实现完整 RBAC（角色权限系统）或多管理员体系。
- 不在本次改动中实现管理员密码的后端动态配置与加密托管。
- 不扩展聊天主流程或模型回复逻辑，只关注后台授权操作闭环。

## Decisions

1. **新增独立视图 `ADMIN_PANEL`，沿用 App 现有视图切换模式**
   - 方案：在 `View` 类型中新增后台视图，沿用 `Sidebar` 的 `onNavigate` 流程切换。
   - 原因：当前项目已经基于单页状态驱动（`CHAT / SETTINGS / PROFILE`），增量最小且可快速接入。
   - 备选：引入 React Router。放弃原因：当前项目未使用路由库，引入成本高、改动面大。

2. **管理员闸口使用前端会话态 + 默认密码常量**
   - 方案：在后台页内部先展示“管理员密码输入卡片”，验证通过后渲染授权表单。
   - 原因：需求明确默认密码固定，且目标是先上线可用界面；前端会话态可快速达成。
   - 备选：后端新增管理员验证接口。放弃原因：当前需求优先前端落地，不阻塞界面交付。

3. **授权表单复用 `Input`、`Button` 与现有玻璃卡片样式**
   - 方案：字段包含 `username` 与 `quota`，`quota` 默认 `10`，仅允许正整数。
   - 原因：复用现有组件可保证视觉一致并降低维护成本。
   - 备选：新建独立控件库组件。放弃原因：本次场景简单，不需额外抽象。

4. **服务层新增后台授权 API 封装，UI 不直接调用 fetch**
   - 方案：在 `services/` 内新增 admin service，复用现有 `apiClient` 统一错误处理与 token 注入。
   - 原因：符合当前项目“业务服务层 + 统一 HTTP 客户端”结构，利于后续替换真实接口。
   - 备选：在组件内直接发请求。放弃原因：不利于测试与后续维护。

## Risks / Trade-offs

- **[风险] 默认管理员密码写在前端存在泄露风险** → **缓解**：本次先满足交付，后续切换后端验证接口并移除前端明文密码。
- **[风险] 如果授权 API 尚未上线，提交会失败** → **缓解**：提供明确错误提示，并允许先走占位成功反馈模式（可配置开关）。
- **[风险] 仅前端闸口不能提供真正安全隔离** → **缓解**：在需求文档明确其为过渡方案，并将后端校验列入后续迭代。
- **[风险] 新视图入口增多可能影响侧边栏可用性** → **缓解**：沿用现有导航信息架构，不改变已有入口顺序，仅新增一个后台项。

## Migration Plan

1. 新增后台视图类型与页面组件骨架，接入侧边栏导航。
2. 实现管理员密码闸口与授权表单校验。
3. 新增服务层方法并接入表单提交。
4. 本地验证：未输入/错误密码、默认调用次数、非法配额、提交成功与失败提示。
5. 回滚策略：若上线异常，可暂时隐藏后台入口并保留代码（低风险回退）。

## Open Questions

- 后台授权接口的最终路径与请求体字段名是否固定（例如 `username` + `quota`）？
- 提交成功后是否需要展示历史授权记录或支持批量授权？
- 管理员密码是否需要支持环境变量配置（而不是硬编码默认值）？
