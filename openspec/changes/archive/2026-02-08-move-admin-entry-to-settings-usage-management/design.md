## Context

当前实现中，“后台管理”入口位于侧边栏，与聊天导航（新建会话、会话列表）混排，导致信息架构上管理能力与对话能力层级混杂。系统已经具备独立的管理页面（含管理员密码闸口与授权表单），因此本次设计重点是调整入口位置和命名，不改动既有授权业务逻辑与接口调用流程。

## Goals / Non-Goals

**Goals:**
- 将管理入口从侧边栏迁移到设置页面，入口名称统一为“用量管理”。
- 保持现有视觉风格一致，复用 Settings 页现有卡片风格和交互模式。
- 保持 AdminPanel 的管理员校验、表单校验、服务提交逻辑不变，确保改动范围可控。
- 确保用户从“设置 -> 用量管理”可以稳定进入现有管理面板视图。

**Non-Goals:**
- 不重构页面路由体系（仍沿用当前 `View` 状态切换）。
- 不改动后端授权接口、请求参数或闸口密码机制。
- 不新增历史记录、批量授权等用量管理扩展功能。

## Decisions

1. **入口迁移到 Settings 组件，由 Settings 通过回调触发视图切换**
   - 决策：为 `Settings` 增加跳转回调（如 `onNavigateToUsageManagement`），在设置页面渲染“用量管理”入口卡片/按钮，点击后切到 `ADMIN` 视图。
   - 原因：设置页天然承载系统配置入口，符合用户心智且能避免侧边栏主导航过载。
   - 备选方案：在侧边栏保留入口并在设置中新增重复入口。放弃原因：会造成入口重复和维护成本增加。

2. **侧边栏移除后台入口及相关视觉元素**
   - 决策：删除侧边栏“后台管理”按钮及 `ShieldCheck` 图标依赖，保留聊天相关导航内容。
   - 原因：明确导航分层，减少误触和视觉干扰。
   - 备选方案：仅改名不迁移。放弃原因：仍无法解决入口层级不合理问题。

3. **“用量管理”仅作为入口文案统一，不改授权页面核心流程**
   - 决策：入口文案、设置项文案统一为“用量管理”；管理页面内部可逐步同步标题文案，但不改变密码闸口/授权表单流程。
   - 原因：满足本次交互目标的同时降低回归风险。
   - 备选方案：同步重构 AdminPanel 所有术语与结构。放弃原因：会引入不必要的额外变更。

4. **保持现有 View 枚举与渲染分支，不引入新路由**
   - 决策：继续使用 `currentView === 'ADMIN'` 的渲染机制，仅调整入口触发来源。
   - 原因：当前项目已稳定使用状态切换方式，改动最小、验证成本低。
   - 备选方案：引入路由库做设置子页面。放弃原因：超出本次范围。

## Risks / Trade-offs

- **[风险] 用户已习惯侧边栏入口，迁移后短期可能找不到入口** → **缓解**：在设置中使用明显卡片样式与“用量管理”描述文案，强化可发现性。
- **[风险] Settings 新增回调可能影响现有组件调用处** → **缓解**：采用可选回调参数或在 `App.tsx` 唯一接入点补齐并做编译验证。
- **[风险] 文案替换不完整导致页面术语不一致** → **缓解**：提交前检查 Sidebar/Settings/AdminPanel 相关文案并统一。

## Migration Plan

1. 在 `Settings` 组件新增“用量管理”入口 UI 与回调参数。
2. 在 `App.tsx` 将该回调绑定到 `setCurrentView('ADMIN')`。
3. 在 `Sidebar` 移除原后台入口及相关图标依赖。
4. 自测入口链路：登录后进入设置，点击“用量管理”可进入管理页；管理员闸口与授权提交可正常使用。
5. 回滚策略：若出现入口不可达，可临时恢复侧边栏入口并保留设置入口代码。
